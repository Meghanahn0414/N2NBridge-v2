System architecture
Central Platform & mobile front-end
A single multi-tenant Central Platform — eight modules behind one API — serving the citizen mobile app and the representative & admin surfaces. This document maps the modules, the data flow, the mobile client, and the API contract between them.
System at a glance
Every client — the citizen mobile app, the representative dashboard, and the admin portal — talks to one Central Platform through a single authenticated API gateway. The platform is a modular monolith: eight cohesive modules share one codebase and one database, isolated by tenant. External delivery (push, payments) sits outside the trust boundary.

Multi-tenancy model: a shared database with a tenant_id on every row and Postgres row-level security. Directory (Tenant Registry) is the source of truth for which tenants exist and their config; Tenant Provisioning creates a tenant's defaults, roles, and seed data on onboarding; the API Gateway resolves the caller's tenant from the JWT and scopes every query. One citizen (name + mobile) in the Citizen Registry can be linked to more than one tenant without duplicating the person.
Mobile front-end
A single cross-platform codebase (React Native → iOS + Android). Screens sit on a thin service layer that owns auth tokens, the API client, push registration, and an offline cache — so the UI never talks to the network directly.

Push loop: on login the app registers its device token via POST /devices. When a case status changes, the Notifications module pushes through FCM (Android) and APNs (iOS) to that token — so "Ward 7 assigned a caseworker" reaches the citizen without the app polling.
